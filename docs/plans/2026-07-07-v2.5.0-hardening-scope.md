## Tier 1 — Blocks the tag (security / silent-failure)

### Issue 1: `--sanitize` fails to redact `Authorization: Bearer` tokens
**File:** `scripts/backup.sh`, `sanitize_claude_json()`, lines ~644–656 (jq filter)

**Problem:** The `--header` arg passthrough (`elif startswith("--header") then .`) keeps single-arg `--header=Authorization: Bearer <token>` verbatim. The two-arg value form is checked with `test("^(authorization:|Bearer )")`, which is case-sensitive and misses the real-world capitalization `Authorization:`. Net effect: the most common MCP remote-auth pattern ships in cleartext through the one feature whose entire purpose is credential removal.

**Fix:**
- Make the auth-header test case-insensitive: `test("^(authorization:|bearer )"; "i")`.
- Redact the `--header=...` single-arg form too — split on the first `:` or `=` and replace the value, don't keep it whole.
- Add a jq-testable fixture covering both header shapes.

**Acceptance criteria:** A synthetic `claude.json` with both `["--header", "Authorization: Bearer sk-test123"]` and `["--header=Authorization: Bearer sk-test123"]` produces `<AUTH_TOKEN>` (or equivalent placeholder) in the sanitized output, not the original token, in both arg shapes and both cases (`Authorization`/`authorization`).

### Issue 2: `git add ... || true` can silently no-op every scheduled backup
**File:** `scripts/backup.sh`, lines ~1020–1024 (staging loop before `git diff --cached`)

**Problem:** `git add "$BACKUP_DIR/$dir" 2>/dev/null || true` swallows all staging failures. A stale `.git/index.lock` (crash, power loss) makes every future `git add` fail; the script then sees an empty staged diff, logs "No changes detected," and exits 0 — indistinguishable from a genuinely clean run. This is the exact silent-failure class the v2.2.0 incident and the entire Windows observability layer exist to prevent, but it happens *inside* git, downstream of where `bash -n` or exit-code checks would catch it.

**Fix:** Remove the `|| true` swallow. Let a `git add` failure propagate (the script already runs under `set -e`), or explicitly catch it and `log_error` + `exit 1` with a message naming the failing directory. The legitimate case where a `stage_dirs` entry doesn't exist yet is already guarded by a `[ -d "$BACKUP_DIR/$dir" ]` check one line above, so it's safe to fail loud here.

**Acceptance criteria:** Manually create `$BACKUP_DIR/.git/index.lock`, run `backup.sh`, confirm it now exits non-zero with a clear error instead of exiting 0 with "No changes detected." Remove the lock file, confirm normal operation resumes.

## Tier 2 — Should fix before tag (restore-fidelity / correctness)

### Issue 3: `restore.sh` never restores `plugins/data/` or root scripts backed up by `backup.sh`
**File:** `scripts/restore.sh`, `restore_plugins()` (~290–314) and `restore_global_settings()` (~194–245)

**Problem:** `backup_global_settings()` captures `~/.claude/*.{cmd,ps1,js,sh,py}` into `global/`, and `backup_plugin_data()` captures `~/.claude/plugins/data/` — both documented in the README's What-to-Backup table — but nothing in `restore.sh` writes either back. A full restore after a machine rebuild silently drops both categories.

**Fix:**
- In `restore_global_settings()`, add a loop over `cmd ps1 js sh py` extensions mirroring the backup-side loop, restoring `global/*.<ext>` → `$CLAUDE_DIR/*.<ext>`.
- In `restore_plugins()`, after the registry-file loop, add a recursive restore of `$BACKUP_DIR/plugins/data/` → `$CLAUDE_DIR/plugins/data/` (mirror the `restore_user_content` walk pattern for nested dirs).

**Acceptance criteria:** Create a fake `global/foo.sh` and `plugins/data/some-plugin/state.json` in a test backup dir, run `restore.sh --dry-run`, confirm both show as NEW; run for real, confirm both land at the correct destination paths.

### Issue 4: `backup-config.json` and `.gitignore` are never committed by either script
**File:** `scripts/init.sh` (no git add/commit anywhere), `scripts/backup.sh` staging list (~1016)

**Problem:** Following the documented Quick Start exactly (`init.sh` → `backup.sh` → `git push`), the pushed repo never includes `backup-config.json` or `.gitignore` — `backup.sh`'s `stage_dirs` array only covers data directories. On a fresh clone, `restore.sh` hard-fails with "Config file not found... This does not appear to be a valid backup repository," discovered only at disaster-recovery time.

**Fix:** Add `backup-config.json` and `.gitignore` to the files `backup.sh` stages. Prefer explicit existence checks over a blind `|| true` swallow (see Issue 2 — don't reintroduce the same failure-hiding pattern here). Do not have `init.sh` auto-commit (keep the user in control of the first commit); just make sure the very next `backup.sh` run picks the config file up.

**Acceptance criteria:** Fresh `init.sh` + `backup.sh` run, then `git status` in the backup dir shows `backup-config.json` and `.gitignore` as committed (not untracked).

### Issue 5: `restore.sh --dry-run` creates real directories despite "writes nothing" contract
**File:** `scripts/restore.sh`, `restore_plugins()` line ~301, `restore_user_content()` lines ~330/337, `restore_todos()` line ~357

**Problem:** These three functions call `mkdir -p` unconditionally; `restore_skills()` and `restore_projects()` correctly guard the same call with `[ "$DRY_RUN" != "true" ]`. Dry-run on a pristine machine leaves behind empty `~/.claude/plugins/`, `~/.claude/todos/`, and user-content directory trees, contradicting the documented "Nothing is written" guarantee.

**Fix:** Add the same `[ "$DRY_RUN" != "true" ]` guard around every `mkdir -p` in these three functions (four call sites total, including the per-file `mkdir -p "$(dirname "$dest_file")"` inside `restore_user_content`'s loop).

**Acceptance criteria:** On a machine/dir with no `~/.claude/plugins`, `~/.claude/todos`, `~/.claude/plans`, run `restore.sh . --dry-run`, then confirm via `ls` that none of those directories were created.

### Issue 6: Stale `scriptVersion` in Windows wrapper state file
**File:** `scripts/windows/backup-wrapper.ps1`, line ~106

**Problem:** Hardcoded `scriptVersion = '2.4.0'` while all three bash scripts and the README report 2.5.0. `last-run.json` will report the wrong version for every run going forward.

**Fix:** Update the literal to `'2.5.0'`. Consider (separate, optional follow-up, not part of this issue) parsing it from `backup.sh`'s `SCRIPT_VERSION=` line instead of hardcoding, to prevent this drift recurring at the next release.

**Acceptance criteria:** `last-run.json` after a run shows `"scriptVersion": "2.5.0"`.

## Tier 3 — Worth fixing, not release-blocking

### Issue 7: Config parsing edge cases in v2.5.0 pattern resolution
**File:** `scripts/backup.sh`, `parse_config()` lines ~115–146

- Absent `projects` key in config is treated identically to `[]` (silent "back up nothing") instead of defaulting to `["*"]` like every other field defaults. Fix: `jq -r '(.projects // ["*"])[]?'`.
- Unquoted `$pattern` in the glob loop (`for dir in "$CLAUDE_DIR/projects/"$pattern`) word-splits on spaces before globbing — a typo'd pattern with a space could glob against the current working directory. Fix: `local IFS=` scoped around the loop, or build the glob via an array.

**Acceptance criteria:** A config with no `projects` key backs up everything (matches `["*"]` behavior). A pattern containing a space either errors clearly or is treated as a single literal token, never expands against an unrelated directory.

### Issue 8: `--sanitize` scope gaps beyond the header bug (Issue 1)
**File:** `scripts/backup.sh`, `copy_safe_files()` (~737–749), `sanitize_settings()` (~685–711)

- `plugins/` is copied wholesale into the export, including `plugins/data/` (arbitrary plugin state added in v2.2.0) — the "always safe" comment predates that addition and was never re-verified.
- `settings.json`'s `env` block and non-MCP `permissions.allow` entries (e.g. `Bash(ssh <hostname>:*)`) pass through the export unredacted and unflagged.

**Fix:** Exclude `plugins/data/` from `copy_safe_files` (or explicitly document it as reviewed-safe if confirmed plugin data never contains secrets). Redact `settings.json`'s top-level `env` map the same way MCP `env` is redacted in `sanitize_claude_json`. Add non-MCP `Bash(...)` allow-rules to the export README's "review before sharing" list.

**Acceptance criteria:** A synthetic `plugins/data/` dir is absent from a `--sanitize` export unless explicitly opted in. A synthetic `settings.json` with an `env` value is redacted in the export.

### Issue 9: Restore prompt count drift
**File:** `scripts/restore.sh`, line ~588 (`"Projects (${#PROJECTS[@]} configured)"`)

**Problem:** With v2.5.0's pattern-based config, `PROJECTS` in restore.sh now reflects the *raw config patterns* (e.g. `["*"]` → count 1), while the prompt implies a count of actual projects that will be restored — misleading now that one pattern can expand to dozens of directories.

**Fix:** Count backup-dir project subdirectories at prompt time instead of `${#PROJECTS[@]}` (restore.sh already has this logic later in `restore_projects()` — hoist or duplicate the count for the prompt).

**Acceptance criteria:** Prompt text reflects the actual number of project directories present in the backup, not the pattern count.

## Explicitly out of scope — needs a decision, not a silent fix

### Decision 1: Push-failure exit code semantics
**File:** `scripts/backup.sh`, lines ~1035–1044

Currently a `git push` failure logs an error but the script still exits 0 (documented at README line ~322). This predates the v2.3+ observability layer, whose entire purpose is a truthful `LastTaskResult` — as-is, an expired SSH key or dead remote means every run toasts success and the watchdog stays quiet while the off-machine copy (the actual disaster-recovery target) silently ages out of sync. Making this exit non-zero when `git_auto_push: true` is more correct but is a documented-behavior change — needs sign-off before implementing.

### Decision 2: macOS bash 3.2 incompatibility
**File:** `scripts/backup.sh` lines 44, 123 (`declare -A` / `local -A`)

Associative arrays (used since v2.2.0, and now also by the new `seen` dedup in the v2.5.0 glob-resolution loop) require bash ≥4. Stock macOS ships bash 3.2, where the script fails to parse at all. The README's "Cross-platform: Windows Git Bash, macOS, Linux" claim (and QUICKSTART's macOS cron instructions) is currently false for anyone without `brew install bash`. Options: (a) require and document bash 4+ explicitly with a version-check guard and a clear error message, or (b) rewrite the associative-array usages (dedup, per-category counters) without them. Needs a decision on which direction, since (b) is a larger refactor than a bug fix.
